## 0. 重要前置说明 — 这是重塑,不是升级

本 spec 是对原平台的**整体重塑**,不是 §6.1「删 v1 加 v2 cache」那种增量改动。

**核心变化矩阵**:

| 维度 | 旧平台(2026-06-24 + 2026-06-25) | 新平台(本 spec) |
|---|---|---|
| 产品定位 | 5 行业价值流转 Sankey 图 | **行业间相互支撑的知识图谱** |
| 节点数 | 5 个行业 + 各 5-12 子节点(~60 节点) | **GB/T 4754 中类 100+ 节点** |
| 边模型 | 价值流转(value: 数值) | **关系(支撑/依赖/提供/服务/消费,定性)** |
| 数据来源 | 国家统计局真实数据(cn-stats + 硬编码) | **纯 LLM 生成**(不再接任何外部数据源) |
| 边权重 | 真实金额(亿元/万亿元) | **LLM 估的 1-5 强度(定性)** |
| 视觉 | d3-sankey | **react-flow(force layout)** |
| 单位 | 元/人/机构(每行业不同) | **无单位**(定性) |
| 时间维度 | 2024 年单年快照 | **无时间维度(当下)** |
| Footer | 数据来源 + 单位 + 年份 | **无 Footer(无"数据来源"概念)** |

**结论**:
- `2026-06-25-接入真实数据-design.md` 的所有内容(cn-stats / 统计局 / 金融新叙事 / 子色阶 / source_url / E2E-004~006)**整体作废**

## 1. 目标(Goals)

1. **产品定位升级**:"行业价值流转可视化平台" → **"行业价值流转知识图谱"**。
   旧名强调「可视化」,新名强调「知识图谱」 — 价值流转只是图谱的一个视角,本质是「行业之间如何相互支撑」的认知工具。
2. **核心抽象**:**行业知识图谱** — 100+ 个 GB/T 4754 中类作为节点,行业间的「支撑 / 依赖 / 提供 / 服务 / 消费」关系作为边。
3. **数据驱动方式**:**纯 LLM 驱动初始化**(方案 B)。
   - 启动时调 LLM 一次性生成完整的知识图谱(nodes + edges + descriptions + explanations),写入 SQLite
   - 后续不调 LLM,仅当用户点击「AI 解释」按钮时实时调 LLM 做补充解释
   - 不再依赖任何外部数据源(统计局 / cn-stats / 行业协会 / 公开网页)
4. **目标用户**:行业研究 / 商业洞察 / 投研演示 — 让用户在 1 分钟内看清「矿业 → 制造」「能源 → 所有行业」这种跨行业支撑关系。
5. **解决什么问题**:行业间依赖关系通常散落在不同行业报告里、不直观;本平台把 100+ 行业的关系一张图呈现,适合作为商业洞察的"鸟瞰地图"。

---

## 2. 非目标(Non-Goals)

- ❌ **不做量化数据** — 边不是"金额",是"关系强度"(weight 1-5,定性)。**不**查"X 行业给 Y 行业供了多少亿原材料"。
- ❌ **不做时间序列** — 知识图谱是当下快照,无历史 / 无趋势。
- ❌ **不做实时数据刷新** — 初始化结果缓存 7 天,过期才重新生成;无 cron。
- ❌ **不做用户系统** — 仍为本地单人使用,沿用原 MVP 设定。
- ❌ **不做关系的人工编辑界面** — LLM 生成的图就是最终形态。如需修订,**只能**通过"重新初始化"(清缓存 + 重启)。理由:本平台本质是"LLM 认知的快照",不是"知识库管理工具"。
- ❌ **不做多语言** — 全中文(沿用原 spec 设定)。
- ❌ **不做导出**(图片 / PDF / 节点清单)— 留 backlog。
- ❌ **不做节点推荐 / 推荐路径** — 用户自行浏览和筛选。
- ❌ **不做与外部数据源对接** — 纯 LLM 驱动,不接统计局 / cn-stats / 行业协会 / 任何公开网页。

## 3. 数据模型

### 3.1 节点(Node)

代表一个 **GB/T 4754 中类**(中等粒度行业,共 100+ 个)。

```python
class GraphNode(BaseModel):
    id: str                       # GB/T 4754 中类代码,例 "C17"(纺织业)
    label: str                    # 中文名,例 "纺织业"
    category: str                 # GB/T 4754 大类(单字母),例 "C"(制造业)— 用于配色分组
    description: str              # LLM 生成的一句话描述(初始化时一次性生成)
```

**GB/T 4754 大类枚举(用于配色)**:

| 代号 | 名称 | 代号 | 名称 |
|---|---|---|---|
| A | 农、林、牧、渔业 | I | 交通运输、仓储和邮政业(注:不同版本,I/J 顺序可能不同) |
| B | 采矿业 | J | 金融业 |
| C | 制造业 | K | 房地产业 |
| D | 电力、热力、燃气及水生产和供应业 | L | 租赁和商务服务业 |
| E | 建筑业 | M | 科学研究和技术服务业 |
| F | 批发和零售业 | N | 居民服务、修理和其他服务业 |
| G | 住宿和餐饮业 | O | 卫生和社会工作 |
| H | 信息传输、软件和信息技术服务业 | P | 文化、体育和娱乐业 |
| | | Q | 公共管理、社会保障和社会组织 |
| | | R | 国际组织 |

**注**:GB/T 4754-2017 实际有 20 个门类(A-T),本 spec 取最常见的 20 类,精确到 GB/T 4754 标准实际发布版本由实施时确认。

### 3.2 边(Edge)

代表两个中类之间的**定性支撑关系**。

```python
class GraphEdge(BaseModel):
    source: str                   # 起点节点 id(GB/T 4754 中类代码)
    target: str                   # 终点节点 id(GB/T 4754 中类代码)
    relation_type: RelationType   # 支撑 / 依赖 / 提供 / 服务 / 消费
    weight: int = Field(ge=1, le=5)  # 关系强度 1-5,LLM 估计(定性)
    explanation: str              # LLM 生成的一句话解释(初始化时一次性生成)
```

### 3.3 关系类型(RelationType)枚举与语义

| 值 | 中文名 | 语义 | 箭头方向 |
|---|---|---|---|
| `provide` | 支撑 | A 的产品是 B 的原料 / 中间品 | A → B |
| `rely_on` | 依赖 | B 的运营 / 设备 / 流程依赖 A 的输出 | A → B(语义:A 被 B 依赖) |
| `service` | 服务 | A 向 B 提供服务(物流 / 咨询 / IT 等) | A → B |
| `consume` | 消费 | A 消费 B 的产品 / 服务(终端市场) | A → B |

**示例**:

- 纺织业(C17)`provide` → 服装鞋帽针纺织品类制造业(C18):布料是服装的原料 → 箭头 C17 → C18
- 汽车制造业(C36)`rely_on` 钢铁冶炼(C31):造车需要钢材 → 箭头 C31 → C36(rely_on 语义为 C36 依赖 C31)
- 软件开发(I65)`service` → 金融业(J66):银行买 IT 系统 → 箭头 I65 → J66
- 居民消费(O80)`consume` 食品制造业(C13):人吃加工食品 → 箭头 C13 → O80

**设计原则**:
- 箭头方向 = **价值 / 服务的实际流转方向**(原料 → 加工 → 终端消费)
- `rely_on` 字段名表示**被依赖**,箭头方向与"依赖方"相反(等价于"提供"但语义强调"不可或缺")

### 3.4 整体图(Graph)

```python
class KnowledgeGraph(BaseModel):
    nodes: list[GraphNode]                         # 100+ 节点
    edges: list[GraphEdge]                         # 估计 200-500 条边
    generated_at: datetime                         # LLM 生成时间(UTC)
    llm_config_hash: str                           # 生成时的 LLM config hash(provider+model+base_url+api_key 前 8 位)
    schema_version: Literal["v1"] = "v1"           # 图 schema 版本(给未来做不兼容升级用)
```

**注**:`generated_at` 和 `llm_config_hash` 用于决定是否复用缓存(见 §7)。

### 4.2 初始化流程(启动时)

```
[FastAPI lifespan startup]
   ↓
[GraphService.ensure_graph()]
   ↓
1. 计算 llm_config_hash = hash(provider + model + base_url + api_key_prefix)
2. cache_key = f"graph:v1:{llm_config_hash}"
3. 查 SQLite cache:
   ├─ 命中 + 未过期(7 天)→ 直接返回,跳过
   └─ 未命中 / 过期 → 进入 4
4. 调 GraphService._generate_via_llm():
   ├─ 构造 init prompt(§5.1)
   ├─ LLMProviderRegistry.get_active().generate(prompt) → raw_text
   ├─ JSON 解析 + 校验(§5.3)
   │   ├─ 失败 → 重试 1 次(更严 prompt)
   │   └─ 仍失败 → 抛 GraphInitError(向上冒泡)
   ├─ 写 SQLite cache(set + expires_at = now + 7d)
   └─ 返回 KnowledgeGraph
5. LLM 异常(网络/限流/鉴权)→ 不缓存,启动报错 → 前端显示"请配置 LLM"
```

### 4.3 查询时 LLM 角色

| 用户操作 | LLM 角色 | 数据来源 |
|---|---|---|
| 访问首页 | 不调 LLM(用缓存的图) | SQLite |
| 点击节点 → NodePanel | 不调 LLM,显示节点的入边 + 出边 + description | SQLite |
| 点击边 → EdgePanel | 不调 LLM,显示两端节点 + 已有 explanation | SQLite |
| 点击 "AI 解释" 按钮(在 EdgePanel 内) | **实时调 LLM**,生成 2-3 句补充说明 | 实时 LLM,响应式返回 |
| 切换 LLM provider | 触发重新初始化(下个 startup) | 清旧 cache,走 §4.2 流程 |

**设计决策**:
- 解释文本**不存**(只存初始化时 explanation 字段;用户点"AI 解释"产生的补充说明不存,避免 LLM 输出长期占库且每次都不同)
- 实时解释的 rate limit:每分钟 5 次(防止用户狂点把 token 烧光)

### 4.4 关键模块职责

| 模块 | 职责 | 关键接口 |
|---|---|---|
| `GraphService` | 初始化 / 重新初始化 / 解释 | `ensure_graph() -> KnowledgeGraph`<br>`explain_edge(s, t) -> str` |
| `GraphValidator` | GB/T 4754 代码校验 + 边完整性校验 | `validate_nodes(nodes) -> None`<br>`validate_edges(edges, nodes) -> None` |
| `GraphCache` | 图缓存读写 | `get(hash) -> KnowledgeGraph \| None`<br>`set(hash, graph)`<br>`invalidate(hash)` |
| `GraphPromptBuilder` | 构造 init prompt / explain prompt | `build_init_prompt() -> str`<br>`build_explain_prompt(s, t) -> str` |

## 5. LLM Prompt 设计

### 5.1 初始化 Prompt(关键)

**系统提示**:

```
你是中国产业研究专家,精通《GB/T 4754 国民经济行业分类》标准。

请基于你的专业知识,生成 100+ 个 GB/T 4754 中类(中等粒度行业)
之间的相互支撑关系知识图谱,作为商业洞察的鸟瞰地图。

要求:
1. 节点(node)覆盖 GB/T 4754 大部分中类(≥ 100 个)
2. 每条边(edge)代表两个行业之间清晰的"支撑 / 依赖 / 提供 / 服务 / 消费"关系
3. 边的箭头方向 = 价值 / 服务的实际流转方向(原料 → 加工 → 终端)
4. weight 是 1-5 的定性强度(1=弱,5=强),无需引用具体数字
5. explanation 一句话说清这条关系(中文,15-30 字)
6. description 一句话介绍节点(中文,15-30 字)
7. 严格按 JSON schema 输出,不要任何额外文字
```

**用户提示**(变量部分,固定):

```
请输出一个 JSON 对象,包含 nodes 和 edges 两个数组,节点 id 必须使用
GB/T 4754 中类代码(如 "C17" 纺织业),category 为单字母大类代码。

JSON schema:
{
  "nodes": [
    {
      "id": "C17",
      "label": "纺织业",
      "category": "C",
      "description": "将纺织纤维加工成纱、线、织物的行业"
    }
  ],
  "edges": [
    {
      "source": "B06",
      "target": "C17",
      "relation_type": "provide",
      "weight": 4,
      "explanation": "煤炭为纺织印染提供热能与动力"
    }
  ]
}

关系类型枚举(必填其一):
- "provide"   支撑(A 的产品是 B 的原料)
- "rely_on"   依赖(B 的运营依赖 A;箭头方向 = A → B,语义: B 依赖 A)
- "service"   服务(A 向 B 提供服务)
- "consume"   消费(A 消费 B 的产品)

weight 强度参考:
- 1: 微弱 / 间接
- 2: 较弱
- 3: 中等
- 4: 较强
- 5: 核心 / 不可或缺
```

### 5.2 实时解释 Prompt(EdgePanel「AI 解释」)

**系统提示**:

```
你是行业研究专家,擅长用 2-3 句话解释两个行业之间的关系。
请基于关系类型和已有解释,生成一段补充说明,让非专业读者也能理解。
```

**用户提示**(变量,模板):

```
源行业:{source_label}({source_id})
目标行业:{target_label}({target_id})
关系类型:{relation_type_cn}
已有解释:{existing_explanation}

请生成 2-3 句补充说明,具体说明:
1. 这条关系在现实经济中的典型场景(举 1-2 个具体例子)
2. 关系强度高/低的主要原因
```

**输出**:纯文本,2-3 句话(50-100 字),不需 JSON 包装。

### 5.3 校验与重试

| 校验项 | 失败行为 |
|---|---|
| LLM 返回非 JSON | 重试 1 次(append "请严格按 JSON 输出,不要任何额外文字") |
| JSON 缺 `nodes` / `edges` 字段 | 重试 1 次 |
| 节点 id 不符合 GB/T 4754 中类代码(非字母+2 位数字) | 静默丢弃非法节点,不报错 |
| 节点 id 不在 GB/T 4754 中类白名单 | 静默丢弃(理由:模型可能幻觉出"X99"这种代码) |
| 边 source / target 不在 nodes 列表 | 静默丢弃该边 |
| `relation_type` 不在枚举 | 静默丢弃该边 |
| `weight` 越界(<1 或 >5) | clamp 到 [1, 5] |
| 节点 < 50 个 | 重试 1 次(prompt append "至少 100 个节点") |
| 边 < 50 条 | 接受(不强求),log warn |
| 仍失败 | 抛 `GraphInitError`,lifespan 捕获后让启动失败,前端首页显示"请配置 LLM" |

**白名单**:`backend/graph/gbt4754_mid_codes.py` 存 100+ 中类代码清单(实施时由人工整理或从公开 GB/T 4754 文档解析,本 spec 不规定具体来源)。

## 7. 缓存策略(适配新用途)

### 7.1 Cache key 设计

**唯一 key**:`graph:v1:{llm_config_hash}`

**`llm_config_hash` 计算**:

```python
import hashlib
config_str = f"{provider}|{model}|{base_url}|{api_key[:8]}"
return hashlib.sha256(config_str.encode()).hexdigest()[:16]
```

**含义**:
- 切换 provider / model / base_url / API key → hash 变 → 重新生成
- 同配置重启 → hash 一致 → 复用缓存(7 天内)
- 不存原始 API key 入 hash,只取前 8 位(防 key 泄露到 cache key 命名)

### 7.2 TTL

| 内容 | TTL | 理由 |
|---|---|---|
| `graph:v1:{hash}` | **7 天** | 初始化结果稳定(同 prompt + 同模型,输出近似),复用省 token |
| 实时 LLM 解释 | **不缓存** | 用户每次点都重新调(刻意设计,见 §4.3) |
| 旧 cache(任何 `*:v*` 前缀) | 启动时全清 | 旧 schema 完全不兼容,无迁移价值 |

### 7.3 LLM 失败的缓存行为

| 场景 | 是否写 cache |
|---|---|
| LLM 调通 + 校验通过 | 写,TTL 7 天 |
| LLM 调通 + JSON 解析失败 + 重试仍失败 | **不写**(避免坏数据污染) |
| LLM 调不通(网络/限流/鉴权) | **不写** |
| LLM 调通 + 节点 < 50 + 重试仍不够 | **不写** |

**理由**:坏数据缓存会让用户在 7 天内都看不到图,且无法通过重启恢复(只能手动清 cache)。

### 7.4 用户重启行为

```
[用户重启服务] → lifespan 启动 → GraphService.ensure_graph()
   ↓
[计算 llm_config_hash] → 查 cache
   ├─ 命中(7 天内)→ 直接用,1ms 返回
   └─ 未命中 → 调 LLM 重新生成(数秒-数十秒,取决于 provider)
       ├─ 成功 → 写 cache
       └─ 失败 → 启动报错(返回非 0 退出码),前端显示"请配置 LLM"
```

## 11. 风险与缓解

| # | 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|---|
| R1 | LLM 生成节点 < 100(覆盖不全 GB/T 4754 中类) | **高** | 图稀薄 | 接受"生成的就行",不强制 100+;最低 50 节点门槛 + 重试 1 次 |

| R7 | LLM 实时解释 token 消耗不可控 | **中** | 成本 | rate limit 5/min;每日 token budget(沿用旧 spec 100k/天) |
| R8 | `rely_on` 箭头方向歧义(读者可能误读) | **中** | 理解偏差 | EdgePanel 加图例说明;hover 提示箭头语义;hover 边时高亮两端节点 |

## 附录 C:本 spec 显式未规定 / 需用户确认的项

> 实施时如发现以下点决策有歧义,应**回到用户确认**而非自行决定。

1. **GB/T 4754 中类白名单数据来源**:本 spec 不规定 `gbt4754_mid_codes.py` 的具体 100+ 节点列表如何获取。
   - 选项 A:人工整理(从 GB/T 4754-2017 文档抄录)
   - 选项 B:从 GB/T 4754 PDF / 公开网页程序化解析
   - 选项 C:让 LLM 自己列出(但 LLM 可能幻觉,需后置校验)
   - **建议**:选项 A(简单 + 权威),实施时给用户最终决策

2. **`rely_on` 箭头方向细节**:本 spec 定义"箭头方向 = A → B,语义:B 依赖 A",与 `provide` 箭头方向相同,仅语义标签不同。
   - 备选方案:`rely_on` 箭头方向反过来(B → A,语义:箭头指向被依赖方)
   - **本 spec 决策**:与 `provide` 方向一致(价值流转方向),用语义标签区分
   - 实施时如 UI 测试发现更易理解的反向,**回到用户确认**

3. **节点 `x` / `y` 坐标**:本 spec 数据模型中 `GraphNode` 不强制存坐标(react-flow dagre 自动算)。
   - 备选方案:存持久化坐标(让用户手动调整位置后保留)
   - **本 spec 决策**:不存(react-flow 算,刷新会重置)
   - 实施时如要支持"位置持久化",回到用户确认是否进 backlog

4. **AI 解释的语言**:本 spec 默认中文,与全站一致。
   - 实施时如要支持英文,**回用户确认**

5. **节点 / 边点击的路由 vs Modal**:本 spec 选路由(URL 可分享,可前进后退)。
   - 备选:Modal(更快但 URL 不留痕)
   - **本 spec 决策**:路由
   - 实施时如要改 Modal,回用户确认

6. **缓存 7 天 TTL 是否可配置**:本 spec 写死 7 天。
   - 备选:Settings 加"图缓存有效期"配置
   - **本 spec 决策**:写死 7 天,简化
   - 实施时如要可配置,回用户确认

7. **图生成的并发安全**:多个 worker / 多用户同时启动的并发问题。
   - **本 spec 决策**:本地单人使用,不做并发控制(单 SQLite 串行)
   - 实施时如要支持多进程 / 多用户,**回用户确认**(可能改架构)

8. **GB/T 4754 标准版本**:本 spec 写 2017 版,实际可能 2025 有新版。
   - 实施时确认当前最新版本,如有变化,中类代码清单需同步更新
